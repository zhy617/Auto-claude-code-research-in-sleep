## 1. Task

- 论文要解决的不是“让 MLLM 更准”，而是一个更具体的系统问题：在 **MoE-based MLLM** 上，面对高分辨率图像产生的大量视觉 token，如何在 **不重新训练** 的前提下，显著降低推理 FLOPs / 延迟，同时尽量保持原模型的多模态理解能力。
- 它关注的是 **推理时加速**，不是训练时压缩；关注的是 **视觉 token 的冗余消除**，不是整个模型结构重设计。

```text
Given:
- 一个已训练好的 MoE-MLLM
- 多模态输入序列 T = [Tv, Tt]
- 其中 Tv 是视觉 token 序列，Tt 是文本 token 序列

Predict / optimize:
- 输出与原模型尽可能一致的回答 y'
- 同时最小化推理计算成本 C

Subject to:
- 不额外训练，不使用额外训练数据
- 方法需可直接插入现有 MoE-MLLM 推理流程
- 主要压缩视觉 token 相关计算，不破坏文本侧语义对齐
- 适用于工业级长视觉序列，而非只在短序列 VLM 上有效

Goal:
- 在性能下降尽量小的情况下，减少视觉 token 数量与视觉 token 激活的专家数
- 即在 Q(y') 约等于 Q(y) 的前提下，让 C(y') 显著小于 C(y)
```

- 更细一点看，论文把任务拆成两个互补子问题：
- 子问题 1：对视觉 token，是否真的每层都需要激活原本那么多专家？
- 子问题 2：在视觉 token 很长时，哪些 token 或 token 窗口在 MoE 路由视角下是冗余的，可以合并或删除？

- 对应的核心优化对象可以写成：

```text
minimize  InferenceCost = f(sequence_length, active_experts_per_token)
subject to PerformanceDrop <= delta
```

- 其中 `sequence_length` 主要由视觉 token 数决定，`active_experts_per_token` 是 MoE 特有的第二个计算杠杆。
- 这篇论文真正的任务定义因此比普通视觉 token pruning 更强：它不是只减少 token，而是同时利用 **MoE 路由结构** 去减少 token 和专家计算。

## 2. Challenge

- 难点 1：高分辨率视觉输入把问题从“几百个 token 的推理”变成了“上千个视觉 token 的工业部署问题”。
- 在这种设置下，注意力和 FFN 计算都会迅速增大，尤其对带动态裁剪的模型，视觉序列长度本身就很夸张。

- 难点 2：现有 pruning 方法主要是为 **dense VLM** 设计的，它们默认“算力瓶颈主要来自 token 多”，但在 MoE-MLLM 中，计算还强烈取决于“每个 token 激活多少专家”。
- 也就是说，dense 模型里常见的思路只看 `token redundancy`，但在 MoE 模型里还存在 `expert redundancy`。前者只砍序列长度，后者可以直接砍专家计算。

- 难点 3：视觉 token 的冗余判断不能只靠 cross-modal attention。
- 注意力低不等于在 MoE 内部路由上也冗余。尤其当 token merging 改写了 token 表征后，可能会连带改变 expert routing，进而破坏 MoE 的专门化分工。
- 论文明确指出，SparseVLM 这类 dense 时代的融合策略在 MoE-MLLM 上会扰动 expert routing，这说明“合并 token”在这里不是无害操作。

- 难点 4：即使要减少激活专家，也不能随便减。
- 如果把对文本 token 的专家激活也一起削减，会明显伤害性能，因为文本 token 更承载推理和语义对齐。
- 如果把低权重专家和高权重专家一视同仁地删，也容易出问题，因为不同专家贡献并不等价。

- 难点 5：在极端压缩下，保留语义细节尤其难，OCR 和细粒度视觉理解最脆弱。
- 这也是为什么单纯粗暴删 token 会在 OCRBench 上掉得很厉害，因为文本区域、局部结构和小目标信息很容易被误删。

- 旧方法为什么不够：
- FastV / SparseVLM 之类方法抓住了“视觉 token 有冗余”这一点，但没有回答“MoE 模型里冗余到底体现在哪一层结构上”。
- 它们没有利用路由分布，没有把“激活专家个数”当成可控变量，也没有针对共享专家 / 非共享专家的结构差异做机制设计。
- 所以旧方法解决的是“dense VLM 的 token 冗余问题”，而这篇论文解决的是“MoE-MLLM 的双重冗余问题：token 冗余 + expert activation 冗余”。

## 3. Inspiration

### 3.1 Explicit Inspiration

- 作者明确观察到：在 MoE 中，**剪掉冗余视觉 token** 从另一个角度看，就是把这些 token 的激活专家数降到 0。
- 这个重解释很关键，因为它把“token pruning”自然连接到了“expert activation reduction”，把 dense 视角的问题转成了 MoE 视角的问题。

- 作者明确观察到：从较浅层之后开始，把视觉 token 的激活专家数直接减半，平均性能几乎不掉。
- 这说明视觉 token 在现有 MoE-MLLM 里并没有充分用满专家容量，存在明显的过度计算。

- 作者明确观察到：低路由权重专家往往贡献更冗余的信息。
- 因此专家不是“均匀有用”的，删后排专家比删前排专家更安全。

- 作者明确观察到：视觉模态的 expert output 存在 **norm concentration / magnitude stability**。
- 也就是说，减少部分专家后，重新归一化得到的融合向量在模长上变化不大，主要只是方向发生小幅偏移，因此性能不容易崩。

- 作者明确观察到：相邻视觉 token 的 routing probability 分布高度相似，且显著高于相邻文本 token。
- 这给了他们一个比 attention 更 MoE-specific 的冗余信号：如果相邻 token 路由到几乎同一批专家，它们大概率在模型内部被视为相近语义片段。

### 3.2 Implicit Inspiration

- 隐式启发 1：既然 MoE 的 router 本质上就在做“把 token 分配给合适专家”的语义分簇，那么 router 输出本身就应该是判断 token 相似性和冗余性的天然信号。
- 换句话说，作者不是额外发明一个新相似度空间，而是直接复用模型内部已经存在的决策信号。

- 隐式启发 2：视觉 token 和文本 token 在模型里的角色不同，因此压缩策略也应该模态不对称。
- 文本 token 更像“任务求解主干”，视觉 token 更像“感知输入缓存”；前者更敏感，后者更有冗余空间。

- 隐式启发 3：冗余窗口不应该统一处理成“全删”。
- 如果一个窗口很冗余但仍有一定重要性，更合理的动作是 merge；如果它既冗余又不重要，才应该 drop。
- 这实际上是在区分“信息重复”与“信息无关”两类情况。

- 隐式启发 4：真正适合工业部署的方法必须 training-free。
- 如果每次为了 pruning 再做校准训练、蒸馏或重训练，那部署收益会被工程成本抵消，尤其不适合大规模已有 MoE-MLLM。

## 4. Insight & Novelty

### 4.1 Core Insights

- Insight 1：在 MoE-MLLM 里，视觉推理的冗余不只体现在“视觉 token 太多”，还体现在“每个视觉 token 激活的专家太多”。
- 类型：systems design + architecture
- 解释：dense pruning 只把计算看成序列长度问题；这篇论文指出，MoE 里还有一个额外自由度，就是每个 token 实际调用多少专家。只要视觉 token 对专家的依赖没那么强，就可以直接削掉一部分专家计算。

- Insight 2：视觉 token 对专家的依赖是“弱集中但高稳定”的，因此保留高权重专家、删除低权重专家通常只会带来有限表示偏移。
- 类型：representation + optimization
- 解释：论文的核心不是简单说“低权重专家不重要”，而是进一步用输出模长稳定性与方向偏移分析说明，为什么删专家后不会立刻破坏语义表示。

- Insight 3：在 MoE 模型里，routing distribution 比纯 attention 更适合作为视觉 token 冗余的判据。
- 类型：inference strategy + architecture
- 解释：如果两个视觉 token 的路由分布几乎一样，说明它们会去调用近似相同的专家子网络，模型内部对它们的“处理路径”高度重合，因此冗余判断应利用这类信号。

- Insight 4：冗余度与重要度是两个不同维度，应该联合建模，而不是单指标决策。
- 类型：inference strategy
- 解释：高路由相似度说明“像”，低跨模态注意力说明“当前问题里不重要”。只有把两者结合，才能区分“该 merge 的冗余信息”和“该直接删的低价值信息”。

- Insight 5：MoE-MLLM 当前训练范式可能在视觉侧存在系统性的专家利用不足。
- 类型：problem framing
- 解释：如果一个模型在不训练的情况下，直接把视觉 token 的激活专家减半仍几乎不掉点，这其实暴露了一个更深层问题：现有 MoE-MLLM 并没有把视觉专家路由学到足够“紧致且必要”。

### 4.2 Novelty Mapping

- [Dense pruning 只减少 token 数，忽略 MoE 中“每 token 激活专家数”这一额外计算来源] -> [受 Insight 1 启发] -> [设计了视觉 token 专属的 expert activation reduction 机制：从指定层 `lv` 开始，仅对视觉 token 把激活专家数从 `K` 降到 `Kv`，文本 token 仍保持完整路由；具体上保留 Top-Kv 路由概率最高的专家，并对其权重重新归一化]

- [随意减少专家会破坏表示，导致精度明显下降] -> [受 Insight 2 启发] -> [设计了基于路由权重的 TopK 专家保留策略，而不是 RandomK 或 MinK；核心机制是优先保留高路由权重专家，因为实验表明低权重专家更冗余、贡献更小]

- [Dense 模型的 attention-based token pruning 无法捕捉 MoE 内部的专家路由冗余] -> [受 Insight 3 启发] -> [设计了 routing-aware token pruning：提取每个视觉 token 的 routing probability 分布，把它当作 token 内部处理路径的表征，并据此衡量相邻视觉 token 的冗余]

- [单 token 级别的独立剪枝既噪声大，也难兼顾局部结构连续性] -> [受 Insight 3 启发] -> [设计了 sliding-window pruning：先把视觉 token 划成窗口，再计算窗口内 routing distribution 的平均余弦相似度，以窗口为单位做冗余决策]

- [只看相似性会误删重要信息，只看注意力又会漏掉 MoE 特有冗余] -> [受 Insight 4 启发] -> [设计了联合冗余评分 `Cv = alpha * routing_similarity - (1 - alpha) * attention_importance`，其中高相似、低注意力的窗口被判为更冗余]

- [把所有冗余窗口都直接删除会造成不可逆信息损失，特别是 OCR 和细粒度区域] -> [受 Insight 4 启发] -> [设计了“merge + drop”两阶段压缩：先对高冗余窗口做 merge 生成 merged token，再对剩余窗口中注意力最低的部分直接删除，从而区分“信息重复”与“信息不重要”]

- [简单平均 merge 后的 token 可能与专家网络期望的表示尺度不匹配] -> [受 Insight 2 启发] -> [在窗口合并时引入 MLERP 风格的归一化融合，使 merged token 的模长更接近原 token 中代表性较强的成员，以减轻对下游 expert routing / FFN 的扰动]

- [现有工作没有系统解释为什么视觉 token 可以无训练地减少专家激活] -> [受 Insight 2 和 Insight 5 启发] -> [给出了理论分析与实证支撑：一方面用模长稳定性、近似正交与方向偏移上界解释“为何减专家后性能仍稳”；另一方面用注意力相关性和隐藏状态相似度验证跨模态对齐没有明显崩]

- 如果只看“新模块名字”，这篇论文像是在做两件事；但从第一性原理看，真正的核心创新只有一个：
- **把视觉冗余从 dense 时代的 attention/token 视角，重构成了 MoE 时代的 routing/expert 视角。**
- 其他设计基本都是围绕这个视角转换展开的工程化落地。

## 5. Potential Flaw

### 5.1 Scenario Limitation

- 局限 1：方法主要建立在“视觉 token 高冗余、相邻 token 路由相似”这一前提上，因此更适合高分辨率、局部连续、视觉 patch 高重复的场景。
- 如果输入是强离散布局、非局部依赖极强、或者视觉 token 排列本身不反映语义邻近性，窗口级相似性假设可能失效。

- 局限 2：方法是为 **已有 MoE-MLLM 推理加速** 设计的，不回答“如何训练出本身就高效的 MoE-MLLM”。
- 它揭示了视觉专家利用不足，但没有反过来利用这个发现去改造训练目标、router regularization 或模态感知专家分工。

- 局限 3：论文的主要验证对象只有两类工业级 MoE-MLLM：InternVL3.5 和 DeepSeek-VL2。
- 虽然这已经比很多论文强，但模型族仍偏少，且二者都属于当前主流视觉-语言架构范式，能否迁移到更不同的视觉 tokenizer、视频模型、音视频多模态模型，证据仍不足。

- 局限 4：真实延迟收益没有和 FLOPs 收益等比例增长。
- 论文自己也承认，在缺少高度优化的 fused MoE kernel 时，减少专家激活的理论收益会被内存带宽和 kernel overhead 吃掉。
- 这意味着方法的“算法收益”是真实的，但“系统收益”仍依赖底层实现成熟度。

### 5.2 Data Pathology

- 病理 1：OCR-heavy、细粒度文字密集图像仍是高风险场景。
- 论文虽然比基线稳，但从表格能看出 OCRBench 对压缩很敏感，说明视觉 token 中有些“看起来冗余”的局部，其实承载关键文本信息。

- 病理 2：缺失模态或文本提示极弱时，attention guidance 可能不稳定。
- 该方法部分依赖“最后一个文本 token 对视觉 token 的注意力”来衡量重要性；如果文本提示本身不够聚焦，重要度估计可能偏移。

- 病理 3：若 router 学得不好，routing similarity 就不再是可靠冗余指标。
- 这篇方法默认 router 输出已经有语义意义；但如果 MoE 路由存在偏置、塌缩、噪声或过度受位置影响，那么“路由相似”未必等于“内容冗余”。

- 病理 4：窗口机制默认局部邻近 token 值得一起判断。
- 对跨区域、跨物体、跨尺度但又共享语义的 token，这个局部窗口划分可能过于僵硬，既可能漏掉远距离冗余，也可能误合并边界区域。

### 5.3 Best Follow-up Paper Direction

- 最值得继续做的新论文方向是：**把 FastMMoE 的推理时观察，反过来变成训练时的模态感知路由学习机制。**
- 具体原因：
- 这篇论文最有价值的发现不是某个 pruning 公式，而是“视觉 token 的专家利用率系统性偏低、文本 token 更依赖专家专门化”。
- 这说明现有 MoE-MLLM 的 router 没有显式建模“不同模态需要不同粒度的专家预算”。

- 一个很自然的 follow-up 是：

```text
训练时显式学习 modality-aware expert budget
- 让视觉 token 学会在简单区域少激活专家、复杂区域多激活专家
- 让文本 token 保持更高且更稳定的专家利用
- 同时把 routing similarity 和 downstream importance 作为辅助监督或正则
```

- 这比继续调 alpha / gamma / window size 更像真正的新论文，因为它抓住了结构性原因，而不是继续做推理启发式调参。

## 6. Motivation

- 之前的方法把视觉加速主要理解成“删掉一些不重要的视觉 token”，但这其实是 dense 模型的思路。
- 可是到了 MoE-MLLM，计算不只来自 token 长度，还来自每个 token 会调用多少专家。

- 那么第一个问题自然是：**如果视觉 token 本身就比文本 token 更冗余，那它们是否也在过度调用专家？**
- 论文的实验回答是：是的，而且这种过度调用大到可以在不训练的情况下直接减半。

- 接着问题变成：**为什么减专家不会明显伤害视觉表示？**
- 因为视觉 token 的低权重专家更像“锦上添花”而不是“不可替代”，同时视觉 expert output 的模长更稳定，所以删掉后主要只是方向轻微偏一点，而不是整段表示塌掉。

- 再往下一个问题是：**如果要继续删视觉 token，应该依据什么判断它们冗余？**
- 只看 attention 不够，因为那还是 dense 视角；MoE 里更本质的信息其实是 routing。
- 如果相邻视觉 token 的路由分布几乎一样，说明它们在模型内部会走相似的专家路径，那它们大概率处理的是相近信息。

- 但又不能把所有相似 token 全删掉，因为“相似”不等于“无用”。
- 所以还要再问一句：**这些相似 token 当前对文本理解重要吗？**
- 于是 routing similarity 负责回答“像不像”，attention 负责回答“重不重要”。

- 最后就得到这篇论文最自然的方案链条：

```text
视觉 token 太多 -> 先不要只想着删 token，先看 MoE 里是否连专家也激活过多
如果视觉 token 的专家调用本来就偏冗余 -> 先减少其激活专家数
如果相邻视觉 token 的路由分布非常接近 -> 它们在 MoE 内部处理路径相似，可以视为候选冗余
如果这些 token 还比较重要 -> merge
如果这些 token 又冗余又不重要 -> drop
这样就能同时压缩 token 长度和专家计算，而且尽量不破坏文本主导的推理过程
```

- 所以这篇论文不是简单地“又做了一个 pruning 方法”，而是在回答一个更本质的问题：
- **在 MoE-MLLM 里，应该沿着模型内部的路由机制来做压缩，而不是沿用 dense VLM 时代的单一 attention 视角。**
