# Qwen3-VL Technical Report

## 1. Task

- 这篇论文要解决的不是单一 benchmark 的视觉问答，而是构建一个通用视觉-语言基础模型，使其能够在统一模型中同时处理文本、图像、视频、长文档、多图、多轮交互、GUI 与 agent 场景，并且不能明显牺牲纯文本能力。
- 更形式化地说：

```text
Given:
- 文本 token 序列 xt
- 图像集合 xi，经过视觉编码后变成视觉 token
- 视频 xv，可视为带时间结构的帧序列并转成视觉 token
- 可交错的长上下文 x = [text, image, video, ...]，长度最高 256K tokens

Predict / optimize:
- 生成回答 y
- 或输出结构化结果（如 OCR 文本、Markdown/HTML 解析、bbox / point / 3D box、时间戳、工具调用轨迹、agent 动作）

Subject to:
- 输入模态混合且顺序交错
- 上下文极长，必须支持跨页、跨图、跨时间片检索与推理
- 视频需要时间对齐
- 训练时既要学多模态，又要尽量保留底层 LLM 的文本能力
- 部署时还要覆盖 dense 与 MoE 两类延迟-质量权衡

Goal:
- 在多模态理解、推理、文档、视频、空间、agent 与文本任务上取得强表现
- 同时保持甚至接近文本专用模型的语言/推理能力
```

- 输入：交错文本、图片、视频、长文档页面、GUI 截图、多图序列，以及训练中构造的结构化监督信号。
- 输出：自然语言答案、长链推理答案、OCR/文档解析结果、grounding 坐标、视频时间戳、GUI 决策与工具调用。
- 优化目标：本质上是统一的 next-token prediction + 指令微调 + 蒸馏 + 强化学习，在不同任务上分别对齐格式正确性、答案正确性、时空定位精度、工具使用质量与偏好一致性。
- 成功标准：模型不仅能“看懂”，还要能在长上下文中稳定检索关键视觉证据、完成跨模态推理，并在纯文本 benchmark 上不被多模态训练明显拖垮。
- 失败标准：视觉能力增强但文本能力退化；长视频/长文档下无法稳定定位证据；时序建模失真；模型只学会表面 OCR/VQA，而没有形成可泛化的 multimodal reasoning。

## 2. Challenge

- 难点 1：模态统一并不等于信息统一。文本是离散 token 序列，图像和视频首先是高维连续信号，必须经过视觉编码、压缩和对齐后才能送入 LLM。这个过程中最容易丢的恰恰是后续推理最关键的细粒度空间、布局与时间信息。
- 难点 2：长上下文多模态比长文本难得多。长文本主要考验记忆与检索；长视频/长文档还要求模型保留时空结构、页面布局和跨模态对应关系。上下文一拉到 256K，任何位置编码失衡、视觉 token 压缩过度、时间表示过稀疏，都会先在长视频和长文档上出问题。
- 难点 3：VLM 常见的训练矛盾是“加视觉能力”和“保文本能力”相互拉扯。若多模态数据配比或损失权重处理不好，模型会出现语言能力回退；若过度保文本能力，多模态又学不起来。
- 难点 4：视频时间是特别难的监督对象。旧做法把时间直接绑到位置编码上，长视频时 position id 会变得稀疏而巨大，模型未必真学会了“几分几秒发生了什么”，只是学到一个脆弱的绝对位置模式。
- 难点 5：多模态能力不是单点能力，而是能力组合。OCR、grounding、空间理解、文档解析、视频理解、agent 决策彼此相关，但监督形式不同，容易出现单任务强、多任务不协同的问题。
- 传统方法的不足主要体现在三类：
- 第一类是架构层面：把视觉信息只压成单层 token 送给 LLM，导致低层细节和高层语义之间的信息瓶颈严重，细粒度感知与语言推理之间对接不牢。
- 第二类是位置/时序层面：原始 MRoPE 将 t/h/w 维度分块，频谱分配不均，长视频时序建模会退化；旧式 T-RoPE/绝对时间编码对长视频尤其不稳。
- 第三类是训练层面：若按样本或按 token 简单混训，文本与多模态样本的贡献很难平衡；后训练如果只做 SFT，模型容易“会说不会想”，或在格式、工具使用、复杂约束遵循上不够稳。

## 3. Inspiration

### 3.1 Explicit Inspiration

- 作者明确受到前作 `Qwen2.5-VL` 的局限启发，尤其是长视频理解中的位置编码问题，以及视频时间对齐方式不够直接的问题。
- 作者明确指出受 `DeepStack` 启发，将视觉 token 注入 LLM 多层，以增强 vision-language alignment，并保留从低层到高层的多级视觉表征。
- 作者明确受到 “thinking with images” 一类工作的启发，认为多模态模型不应只输出答案，还应具备 think -> act -> analyze feedback -> answer 的 agentic 过程能力。
- 作者明确从系统和产品目标出发，把“长上下文 256K”“dense/MoE 同时覆盖”“thinking / non-thinking 双模式”作为设计约束，而非单纯算法炫技。

### 3.2 Implicit Inspiration

- 隐式启发 1：真正限制 VLM 上限的，不只是视觉编码器够不够强，而是视觉证据在进入 LLM 后是否还能以合适层级被持续利用。所以作者把重点放在“跨层传递视觉信息”，而不是单纯堆更大的 ViT。
- 隐式启发 2：视频理解的关键不是给模型一个抽象的“时间轴 embedding”，而是给它一种更接近语言世界、可被 LLM 直接操作的时间标记。因此文本化时间戳本质上是在把时序 grounding 转化成语言可消费的对象。
- 隐式启发 3：多模态 reasoning 的瓶颈往往不在“推理器”本身，而在“题目是否真的需要视觉证据”。所以他们在 Long-CoT 数据中过滤掉那些不用看图也能答对的样本，以避免伪多模态推理。
- 隐式启发 4：如果目标是现实世界 agent，而不是 benchmark 机器，那么训练数据必须显式包含工具调用、GUI、空间关系、长文档、长视频等场景，否则模型只会停留在静态单图问答。
- 隐式启发 5：在大模型时代，很多性能提升来自数据结构和训练流程的重写，而不一定来自全新架构。该报告很大一部分增益其实来自“把监督信号组织成模型真正需要学的形式”。

## 4. Insight & Novelty

### 4.1 Core Insights

- Insight 1：长视频/长文档能力首先受制于位置与时间表示是否稳定，而不是只受上下文窗口长度影响。把窗口扩到 256K 只是必要条件，不解决频谱失衡和时间表达问题，长上下文能力仍然会虚高。
- Insight 2：视觉信息不能只在输入层一次性注入。多模态推理需要低层视觉细节和高层语义在 LLM 内部持续对齐，因此多层注入比单次压缩更有效。
- Insight 3：视频时间最好被表示成 LLM 原生能读懂、能引用、能比较的符号对象，而不是仅仅嵌入到连续位置编码里。
- Insight 4：多模态训练若要不伤文本能力，关键是训练配比与损失重加权，而不是简单把更多图文数据塞进去。
- Insight 5：真正的多模态 reasoning 数据必须保证“视觉必要性”；否则模型会学会文本捷径，benchmark 分数看似高，但能力不扎实。
- Insight 6：若目标是 agentic multimodality，后训练必须从“会回答”扩展到“会分步思考、会调用工具、会根据反馈修正行为”。

### 4.2 Novelty Mapping

- [长视频中位置编码频谱失衡、时空依赖建模不稳] -> [Insight 1：长上下文能力受位置表示质量约束] -> [设计了 enhanced interleaved-MRoPE，将时间 t、水平 h、垂直 w 三类维度交错分布到高低频带，而不是按块切分，以缓解频谱偏置并提升长视频时空建模]
- [视觉 token 经过单次压缩后细粒度信息丢失，语言层难以持续利用视觉证据] -> [Insight 2：视觉信息需要在 LLM 内多层持续注入] -> [设计了基于 DeepStack 的跨层融合：从 ViT 不同层提取多级特征，经专门 merger 投影后分别残差注入前几层 LLM hidden states，强化细粒度感知与语言推理对齐]
- [绝对时间位置编码在长视频中 position id 稀疏且训练成本高，对 fps 分布敏感] -> [Insight 3：时间应被转成 LLM 可直接操作的文本对象] -> [设计了 text-based timestamp，对视频 temporal patch 前缀显式加入如 `<3.0 seconds>` 或 HMS 格式的时间戳，实现更直接的 temporal grounding]
- [文本数据与多模态数据混训时，简单损失累加会让一方主导训练，导致文本能力或多模态能力受损] -> [Insight 4：关键是控制不同样本/模态对梯度的相对贡献] -> [设计了 square-root reweighting，从 per-sample loss 转向 square-root-normalized per-token loss，以更平衡地联合优化 text-only 与 multimodal 学习目标]
- [超长上下文直接训练成本高且容易不稳定] -> [Insight 1：长上下文能力需要分阶段建立] -> [设计了 8K -> 32K -> 256K 的四阶段预训练流程，先做 merger 对齐，再全参训练，再逐步扩窗到 ultra-long context]
- [很多“多模态推理”样本实际上可被文本捷径解掉] -> [Insight 5：只有视觉必要样本才能真正训练视觉推理] -> [在 Long-CoT 冷启动数据中过滤掉无需视觉输入也可由文本模型答对的题目，显式保证多模态必要性]
- [仅靠 SFT 难以让模型形成稳定复杂推理和高质量行为对齐] -> [Insight 6：后训练要把 reasoning、preference、instruction following、tool use 分开强化] -> [设计了三段式后训练：SFT 激活能力，Strong-to-Weak Distillation 向轻量模型迁移推理，再用 Reasoning RL + General RL 做可验证推理与行为纠偏]
- [多模态 agent 容易学会“伪工具使用”，例如只打一轮工具调用来刷奖励] -> [Insight 6：agent 能力需要显式约束过程质量，而不只是最终答案] -> [在 thinking with images 的 RL 中加入 tool-calling reward，并联合答案正确性与多轮推理质量奖励，抑制 reward hacking]
- [文档、GUI、空间、3D、视频、代码等任务若各自独立构造，能力难以迁移] -> [Insight 5：现实任务需要的是结构化、可组合的监督信号] -> [设计了覆盖 OCR、Markdown/HTML 解析、grounding、3D box、GUI 轨迹、搜索轨迹、时空 grounding、长文档 VQA 等大规模数据管线，把多模态能力拆成可训练的中间技能]
- [不同部署场景对延迟和质量需求差异很大] -> [Insight 4：统一方法需要兼顾能力与可部署性] -> [同时提供 dense 2B/4B/8B/32B 与 MoE 30B-A3B/235B-A22B 版本，并分别提供 thinking / non-thinking 模式]

## 5. Potential Flaw

### 5.1 Scenario Limitation

- 这篇报告的目标极大而覆盖面极广，但也因此更像“全栈能力打包报告”，而不是对某一个核心科学问题做严格可证伪研究。结果是：很多提升被同时归因于架构升级、数据重构、训练 recipe、长上下文扩窗和后训练增强，单一因素的因果贡献并不总是被充分隔离。
- 论文虽然给了 DeepStack、Qwen3-ViT、Needle-in-a-Haystack 等消融或分析，但对于 interleaved-MRoPE、text timestamp、square-root reweighting、长 CoT 数据过滤、thinking agent RL 之间的相对贡献，缺少系统的正交 ablation。也就是说，哪些改动是“必要的核心创新”，哪些只是“工程上有帮助”，边界并不完全清楚。
- 许多 benchmark 比较对象是闭源商用模型，而视频评测中作者自己也承认由于 API/资源限制，输入帧数不完全公平。因此一些 SOTA 结论需要谨慎看待。
- 论文宣称模型在很多文本 benchmark 上可比甚至超过纯文本基座，但如果放到更系统的语言长期评测、工具代理稳定性评测和真实复杂工作流中，是否仍能维持“视觉增强但文本不退化”，论文中证据还不够充分。
- 对于真实部署场景中的关键约束，例如推理成本、显存占用、超长视频 latency、tool-augmented agent 的失败恢复机制，论文中未明确说明。

### 5.2 Data Pathology

- 该方法高度依赖大规模合成与伪标注数据。如果 caption、OCR、解析、grounding、agent 轨迹本身含有系统偏差，模型会把这些偏差学成“世界知识”或“行为先验”。
- 多模态推理数据虽然做了视觉必要性过滤，但过滤标准仍建立在已有模型能否“盲答”之上。这种过滤可能去除了明显的伪多模态样本，却未必能完全排除更隐蔽的文本捷径。
- 视频时间戳文本化虽然更直接，但它也可能带来新的 shortcut：模型可能过度依赖显式 timestamp 字符串模板，而不是从视觉变化本身理解事件边界。
- GUI/agent 训练若主要来自合成轨迹和模型自举轨迹，容易在真实交互环境中暴露 distribution shift，例如界面变化、延迟、异常状态、工具返回噪声等。
- 长文档和 OCR 任务的训练数据大规模依赖自动解析与伪标签，这会让模型更擅长“复现训练时的版式分布”，但对极端版式、低资源语言、扫描噪声文档的鲁棒性仍可能不足。

### 5.3 Best Follow-up Paper Direction

- 最值得继续做的新论文方向，不是再堆更多 benchmark，而是研究“多模态长上下文中的证据利用真实性”。
- 一个具体方向是：构造能够区分“真正基于视觉/时序证据推理”与“利用显式时间戳、文本捷径、版式偏差取巧”的可证伪评测与训练框架。
- 为什么这个方向最重要：Qwen3-VL 的很多成功来自把信息显式结构化给模型，例如 timestamp、文档布局、GUI 元数据、合成 CoT。这很有效，但也让模型更容易学到 shortcut。下一步真正关键的是让模型在缺少模板化提示时仍能稳健地从视觉证据中推理。
- 可操作的研究题目可以是：在视频、文档、GUI 三类场景上做 counterfactual evidence testing，系统测量模型到底依赖了哪些证据；再设计 anti-shortcut 的训练或 RL 目标，让模型的推理路径对视觉证据更忠实。

## 6. Motivation

- 之前的方法已经证明，把图像 token 接到 LLM 后面可以得到一个能用的 VLM；但一旦任务变成长视频、长文档、多图、多步 agent，瓶颈就不再是“能不能看见”，而是“视觉证据能否在长链推理里保持可用”。
- 如果长视频表现差，问题真的只是上下文窗口不够吗？未必。也可能是位置编码本身把时间结构表达坏了。那么先修正时空位置表示，比单纯扩窗更有必要。
- 如果模型回答里经常出现看过图却用不上图的现象，那说明视觉信息只在入口被压缩了一次，后续语言层并没有持续消化它。那能不能把不同层级的视觉特征直接送到 LLM 多层里？
- 如果视频 grounding 总是不稳，说明把时间藏在位置编码里太间接了。既然 LLM 本来就擅长处理符号化文本，那为什么不直接把时间写成文本 token，让模型像读普通字符串一样读时间？
- 如果多模态训练一加强，文本能力就掉，那说明训练目标失衡了。于是自然会想到：不是停止多模态训练，而是重新设计不同模态样本对训练的贡献方式。
- 如果希望模型不仅能答题，还能做 agent，就不能只训练最终答案；它必须学会什么时候思考、什么时候调用工具、如何根据反馈修正动作。因此后训练也要从单纯 SFT 走向 reasoning RL 与 tool-integrated RL。
- 顺着这条链走下来，这篇论文的核心想法就很自然：先把长上下文和时空表示打稳，再把视觉证据注入到 LLM 真正会用的位置，再用数据和后训练把“看、想、做”三件事接起来，最终得到一个既能看图看视频，也尽量不丢语言能力的通用多模态基础模型。
